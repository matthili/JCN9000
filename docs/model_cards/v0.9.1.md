> `jass_engine.rules.legal_moves` wie bei Kreuz und Solo (siehe v0.7.1 fuer
> Details). Bodensee nutzt `legal_moves_bodensee`, das intern auf
> `legal_moves` aufsetzt -- damit war Bodensee identisch betroffen.

> Encoding-Version (`bodensee_1.0.0`, 291 Dim) bleibt strukturell gleich,
> aber die Web-App muss die Fixtures neu pinnen.

## Begruendung des Re-Trainings

Identisch zu v0.7.1 / v0.8.1. Bodensee zusaetzlich relevant, weil:

1. Der Bedien-/Stech-Zwang im Bodensee-Spielfluss besonders haeufig
   eintritt -- jeder Spieler hat doppelte Karten-Quelle (Hand + sichtbarer
   Tisch), die Lead-Suit-Verteilung ist breiter, der Stech-Druck hoeher.
2. Eine Encoding-Fixture (`bfix_06`) basierte auf einer Stellung, in der die
   alte Buggy-Legal-Moves zu einer anderen Aktionsmaske fuehrte. Die Fixture
   wurde neu generiert -- die Web-App muss
   `spec/fixtures/bodensee_encoding_fixtures.json` neu uebernehmen, sonst
   schlagen die TypeScript-Encoder-Tests fehl.

## Architektur

Identisch zu v0.9.0: MLP 768/768/384, `MaskBias`, 291-dim State + 36-dim
Mask, Policy + Value. Encoder-Layout dokumentiert in
[`spec/bodensee_state_encoding.md`](../../spec/bodensee_state_encoding.md).

Komplett neu erzeugt nach dem Engine-Fix:

**Datengen-Aufwand:** Bootstrap ~1,1 min (reine Python-Spiellogik), B-mcts1
= 117 min, B-mcts2 = 388 min (~6,5 h). Bodensee ist im Datengen schneller
als Kreuz/Solo, weil die Stiche pro Runde laenger sind, aber weniger Runden
pro Spiel anfallen.

**Variables Spielziel:** 50/50 zwischen 500 und 1000 Punkten (wie v0.9.0).

## Trainingsverfahren

| Parameter | Bootstrap | MCTS-Runde 1 | MCTS-Runde 2 |
|---|---|---|---|
| Warm-Start | nein | Bootstrap | bodensee_mcts1 |
| Lernrate | 1e-3 | 5e-4 | 5e-4 |
| Batch-Groesse | 1024 | 1024 | 1024 |
| Epochen | 30 | 20 | 20 |
| Beste Val-Policy-Acc | ~96 % (Bootstrap reproduziert die Heuristik) | 82,53 % | **84,39 %** |
| Beste Val-Value-MAE | ~0,20 | 0,2540 | **0,2540** |

Auffaellig: Die Val-Policy-Acc liegt bei Bodensee deutlich hoeher als bei
Kreuz (60 %) oder Solo (63 %). Grund: Bodensee hat einen kleineren effektiven
State-Space (2 Spieler statt 4, kuerzere Spiele, keine Weisen). Der MCTS-
Lehrer ist damit leichter zu approximieren.

## Bewertung

Eval-Setup: paired-eval, 1000 Partien, Spielziel-Mix 500/1000, `batched-gpu`.

| Vergleich | v0.9.1 Win-Rate | Gegner Win-Rate |
|---|---|---|
| vs. bodensee_mcts1 (eigener Vorlaeufer) | **53,0 %** | 47,0 % |
| vs. Heuristik | **77,8 %** | 22,2 % |

**Wichtige Beobachtung:** Gegen das eigene MCTS-Vorlaeufer-Modell ist der
Sprung mit 53,0 % vergleichsweise gering -- aehnlich wie bei Solo, wo
mcts1 bereits sehr stark war. Gegen die externe Referenz (Heuristik) liegt
v0.9.1 jedoch 5,2 Prozentpunkte ueber v0.9.0 (72,6 %). Die zweite MCTS-Runde
hat die Politik also nur leicht verschoben, aber in eine objektiv staerkere
Richtung.

**Punktverlauf vs. Heuristik:**

| Metrik | v0.9.1 | Heuristik |
|---|---|---|
| Avg. Score pro Partie | 528,3 | 419,3 |
| Avg. Runden pro Partie | 6,0 | 6,0 |
| Matsch-Rate pro Runde | 1,20 % | 0,05 % |

Matsch-Rate ist 24-mal hoeher als bei der Heuristik -- das Modell spielt
deutlich aggressiver auf den Voll-Matsch.

**Pro Variante** (Win-Rate v0.9.1 vs. Heuristik):

| Variante | Win-Rate | Stichprobe (Runden) |
|---|---|---|
| Trumpf Eichel | 75,0 % | 72 |
| Trumpf Schelle | 67,1 % | 82 |
| Trumpf Herz | 74,1 % | 54 |
| Trumpf Laub | 74,2 % | 62 |
| Gumpf Eichel | 80,2 % | 106 |
| Gumpf Schelle | 79,8 % | 94 |
| Gumpf Herz | 78,9 % | 76 |
| Gumpf Laub | 75,0 % | 92 |
| Oben | 80,3 % | 142 |
| Unten | **81,8 %** | 220 |

Schwaechste: Trumpf Schelle (67,1 %, n=82). Staerkste: Unten (81,8 %, n=220).
Im Vergleich zu v0.9.0 sind die Trumpf-Varianten leicht staerker geworden
(v0.9.0: 70-71 %; v0.9.1: 67-75 %), die Gumpf-Familie deutlich (v0.9.0:
71-75 %; v0.9.1: 75-80 %).

**Variant-Sub-Eval gegen bodensee_mcts1:** Bei den "balancierten" Modi
*Oben* (47,9 % vs 52,1 %) und *Unten* (49,1 % vs 50,9 %) liegt v0.9.1 leicht
unter mcts1 -- in Modi ohne Trumpf-Asymmetrie ist mcts1 schon nahe am
Optimum. Gegen die Heuristik bleiben beide Modi staerkste Disziplinen, also
keine Regression sondern Saettigung.

## Bekannte Schwaechen

- **Slalom kommt im Bodensee nicht vor** (Bodensee-Heuristik waehlt keine
  Slalom-Variante an).
- **`i_am_announcer`-Feature im Encoder** ist im Datengen pragmatisch immer
  auf `False` gesetzt -- das Bit ist faktisch tot. Effekt auf die Spielstaerke
  ist gemessen marginal, aber eine sauberere Implementierung bleibt offen.
- **Mensch-vs-Modell-Validierung** steht aus.
- **Saettigung bei Oben/Unten** (siehe oben) -- weitere MCTS-Runden duerften
  bei diesen Modi keine relevante Verbesserung bringen.

## Reproduktion

WSL2 mit GPU, conda-Env `jass-gpu`:

```bash
# Bootstrap-Daten (20.000 Partien Heuristik vs Heuristik, ~1 min)
python -u -m training.data.generate_bodensee_heuristic_data \
    --games 20000 --games-per-shard 200 --workers 12 \
    --target-distribution "500:0.5,1000:0.5" \
    --output data/bodensee_heuristic_bootstrap

# Bootstrap-Modell
python -u -m training.train \
    --data data/bodensee_heuristic_bootstrap \
    --output models/bodensee_base --epochs 30

# MCTS-Runde 1
python -u -m training.data.generate_bodensee_mcts_data_mp \
    --warm-start models/bodensee_base/best.keras \
    --games-per-variant 150 --games-per-chunk 25 \
    --rollouts-per-card 30 --target-distribution "500:0.5,1000:0.5" \
    --workers 8 --parallel-threads-per-worker 32 \
    --inference-batch-size 1024 \
    --output data/bodensee_mcts_fixed/phase1

python -u -m training.train \
    --data data/bodensee_mcts_fixed/phase1 \
    --warm-start models/bodensee_base/best.keras \
    --output models/bodensee_mcts1 \
    --epochs 20 --learning-rate 5e-4

# MCTS-Runde 2
python -u -m training.data.generate_bodensee_mcts_data_mp \
    --warm-start models/bodensee_mcts1/best.keras \
    --games-per-variant 500 --games-per-chunk 25 \
    --rollouts-per-card 30 --target-distribution "500:0.5,1000:0.5" \
    --workers 8 --parallel-threads-per-worker 32 \
    --inference-batch-size 1024 \
    --output data/bodensee_mcts_fixed/phase2

python -u -m training.train \
    --data data/bodensee_mcts_fixed/phase2 \
    --warm-start models/bodensee_mcts1/best.keras \
    --output models/bodensee_mcts2 \
    --epochs 20 --learning-rate 5e-4

# Eval
python -m evaluation.run_bodensee_eval \
    --a nn --b heuristic \
    --model-a models/bodensee_mcts2/best.keras \
    --games 1000 --paired-eval \
    --inference-mode batched-gpu
```

Encoder-Struktur unveraendert. **Drei Aenderungen** muessen aber synchron in
die App:

1. **Legal-Moves-Fix** in der TypeScript-Engine ("bedienen ODER stechen").

   `bfix_06` aktualisiert). Die TypeScript-Encoder-Tests muessen die neue
   Fixture pinnen.
3. **Neues TF.js-Modell** aus dem v0.9.1-Release-ZIP.

Details: [`docs/web_app_update_v0.9.1.md`](../web_app_update_v0.9.1.md).
