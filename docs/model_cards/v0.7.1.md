# Modell-Karte: Jass-NN v0.7.1 (Kreuz-Jass, Re-Training nach Legal-Moves-Fix)

> **Punkt-Release ueber v0.7.0.** Anlass: Im Kartenausspiel war der
> Bedien-/Stech-Zwang in `jass_engine.rules.legal_moves` falsch implementiert
> (siehe Abschnitt "Begruendung des Re-Trainings"). Architektur, Encoding und
> Training-Hyperparameter sind unveraendert -- die Trainingsdaten wurden
> komplett gegen die korrigierte Engine neu erzeugt.

## Begruendung des Re-Trainings

In `jass_engine.rules.legal_moves` lautete die Regel fuer "Lead-Farbe ist
keine Trumpf-Farbe, ich habe Lead-Farbe auf der Hand":

- **falsch:** "bedienen; zusaetzlich darf nur der Buur (Trumpf-Unter) gespielt
  werden"
- **richtig:** "bedienen ODER stechen -- d. h. alle Lead-Farb-Karten sind
  legal, alle Trumpf-Karten (die nicht untertrumpfen) sind legal, der Buur
  ist immer legal"

Auswirkung: Das Modell hat strategisch wichtige Stech-Optionen nie als legal
gesehen und konnte sie weder lernen noch ausspielen. Korrektur in
`jass_engine/rules.py`, abgesichert durch drei zusaetzliche Regel-Tests in
`tests/test_rules.py` (`test_bedienen_oder_stechen_mit_lead_farbe_und_normalem_trumpf`,
`test_bedienen_oder_stechen_kein_untertrumpfen_mit_lead_farbe`,
`test_gumpf_bedienen_oder_stechen_mit_normalem_trumpf`).

Da Trainingsdaten *und* MCTS-Lookahead die Engine verwenden, mussten saemtliche
Phasen neu durchlaufen werden. Die Daten von v0.7.0 sind deprecated.

## Architektur

Identisch zu v0.7.0 (siehe dort fuer Details): MLP 768/768/384, `MaskBias`,
421-dim State + 36-dim Mask, Policy + Value, TensorFlow/Keras.

## Bewertung

Eval-Setup: paired-eval (pro Kartenverteilung 2 Partien mit gespiegelten
Sitzen, eliminiert Karten-Glueck), je 1000 Partien, Spielziel 1000,
`batched-gpu`.

| Vergleich | v0.7.1 Win-Rate | Gegner Win-Rate |
|---|---|---|
| vs. K-mcts1 (eigener Vorlaeufer) | **71,4 %** | 28,6 % |
| vs. Heuristik | **79,5 %** | 20,4 % |

Zur Einordnung: v0.7.0 erreichte gegen die Heuristik 77,2 %. Trotz korrigierter
Engine (die der Heuristik tendenziell *mehr* legale Optionen gibt) liegt v0.7.1
2,3 Prozentpunkte hoeher. Gegen das eigene MCTS-Vorlaeufer-Modell ist die
71,4 %-Win-Rate ein robustes Lern-Signal aus MCTS-Runde 2.

**Punktverlauf vs. Heuristik:**

| Metrik | v0.7.1 | Heuristik |
|---|---|---|
| Avg. Score pro Partie | 1031,8 | 795,3 |
| Avg. Runden pro Partie | 9,3 | 9,3 |
| Matsch-Rate pro Runde | 4,41 % | 2,15 % |

**Pro Variante** (Win-Rate v0.7.1 vs. Heuristik, je 1000 Partien gesamt):

| Variante | Win-Rate | Stichprobe (Runden) |
|---|---|---|
| Trumpf Eichel | 63,2 % | 924 |
| Trumpf Schelle | 63,9 % | 905 |
| Trumpf Herz | 65,2 % | 858 |
| Trumpf Laub | 63,1 % | 923 |
| Gumpf Eichel | 57,6 % | 792 |
| Gumpf Schelle | 57,9 % | 807 |
| Gumpf Herz | 58,7 % | 861 |
| Gumpf Laub | 58,3 % | 785 |
| Oben | 64,2 % | 215 |
| Unten | **66,4 %** | 503 |
| Slalom Oben | 65,4 % | 856 |
| Slalom Unten | **67,2 %** | 900 |

Wie schon bei v0.7.0: Gumpf bleibt die schwaechste Familie (57-58 %).
Slalom und Unten sind die staerksten -- bei diesen Modi profitiert das Modell
am meisten vom korrigierten Stech-Repertoire.

## Bekannte Schwaechen

Identisch zu v0.7.0 (siehe dort). Zusaetzlich:

- **Erstes Re-Training-Release.** Die paired-eval-Stichprobe ist mit 1000
  Partien kleiner als bei v0.7.0 (4000). Standardabweichung der Win-Rate liegt
  bei ca. 1,6 Prozentpunkten -- die Abstaende bleiben deutlich, aber feinere
  Sub-Varianten-Analysen sind weniger belastbar.

## Reproduktion

WSL2 mit GPU, conda-Env `jass-gpu`:

```bash
# MCTS-Runde 1 -- Warm-Start: v0.5.0-Modell (Heuristik-Aequivalent vor dem Bug)
python -u -m training.data.generate_mcts_data_mp \
    --warm-start models/v5/best.keras \
    --games-per-variant 150 --games-per-chunk 25 \
    --rollouts-per-card 30 --target 1000 \
    --workers 8 --parallel-threads-per-worker 32 \
    --inference-batch-size 1024 \
    --lookahead-mode full-round-vec \
    --output data/mcts_fixed/phase1

python -u -m training.train \
    --data data/mcts_fixed/phase1 \
    --warm-start models/v5/best.keras \
    --output models/kreuz_mcts1 \
    --epochs 20 --learning-rate 5e-4

# MCTS-Runde 2
python -u -m training.data.generate_mcts_data_mp \
    --warm-start models/kreuz_mcts1/best.keras \
    --games-per-variant 500 --games-per-chunk 25 \
    --rollouts-per-card 30 --target 1000 \
    --workers 8 --parallel-threads-per-worker 32 \
    --inference-batch-size 1024 \
    --lookahead-mode full-round-vec \
    --output data/mcts_fixed/phase2

python -u -m training.train \
    --data data/mcts_fixed/phase2 \
    --warm-start models/kreuz_mcts1/best.keras \
    --output models/kreuz_mcts2 \
    --epochs 20 --learning-rate 5e-4

# Eval
python -m evaluation.run_eval \
    --a nn --b nn \
    --model-a models/kreuz_mcts2/best.keras \
    --model-b models/kreuz_mcts1/best.keras \
    --games 1000 --paired-eval \
    --inference-mode batched-gpu \
    --inference-batch-size 128 --parallel-threads 128
```
