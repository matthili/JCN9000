# Modell-Karte: Jass-NN v0.7.0

> Eine Modell-Karte ("Model Card") ist eine kurze, strukturierte Beschreibung
> eines Machine-Learning-Modells, die dokumentiert: Wofuer wurde es trainiert,
> auf welchen Daten, mit welchen Methoden, wo sind die bekannten Schwaechen,
> wie kann man es reproduzieren. Pro Release eine eigene Karte, damit ein
> Entwickler oder Pruefer ohne Code-Wuehlen verstehen kann, was er bekommt.

## Verwendungszweck

Spiel-KI als Gegner fuer menschliche Spieler in der Web-App "Heb ab!"
(separates Projekt). Trifft pro Spielposition eine Karten-Entscheidung
basierend auf der eigenen Hand, der bekannten Spielhistorie und der
gewaehlten Variante. Schaetzt zusaetzlich den erwarteten Spielausgang
(Value-Head), wird aber im aktuellen Web-Einsatz nur fuer die
Karten-Wahl genutzt.

**Nicht gedacht fuer:**
- Pflicht-Trumpf-Ansage (uebernimmt der Heuristik-Bot als Fallback)
- Weis-Ansage (uebernimmt der Heuristik-Bot)
- Live-Coaching / Empfehlungen waehrend einer menschlichen Partie ohne
  klare Kennzeichnung als KI-Vorschlag

## Architektur

| Eigenschaft | Wert |
|---|---|
| Modell-Typ | Mehrschichtiges Perzeptron (MLP) |
| Hidden-Layer | 768 - 768 - 384 (ReLU) |
| Parameter | ~1.25 Mio. |
| Eingabe | `state` (421 Dimensionen, float32) + `mask` (36 Dimensionen, float32) |
| Ausgabe | `policy` (36 Wahrscheinlichkeiten) + `value` (1 Skalar in `[-1, 1]`) |
| Custom-Layer | `MaskBias` (addiert grosses negatives Bias auf illegale Aktionen vor dem Softmax) |
| Framework | TensorFlow 2.21 / Keras 3.14 |
| Browser-Format | TensorFlow.js (~14 MB als ZIP mit zwei Shards) |

## Trainingsdaten

| Quelle | Anzahl Samples |
|---|---|
| MCTS-Phase 1 (Kalibrierung mit v0.5.0 als Lehrer-Init) | 303.912 |
| MCTS-Phase 2 (mit Phase-1-Modell als Lehrer-Init) | 2.095.056 |
| **Gesamt** | **2.398.968 Samples** |

**Datenerzeugung** (siehe [`training/data/generate_mcts_data.py`](../../training/data/generate_mcts_data.py)
und die Multiprocessing-Variante [`generate_mcts_data_mp.py`](../../training/data/generate_mcts_data_mp.py)):

Pro Spielposition wurde ein vereinfachter Monte-Carlo-Lookahead durchgefuehrt:
- Pro legaler Karte ca. 30 Rollouts mit zufaellig verteilten unsichtbaren Karten
  (Determinisierung)
- Bewertung jeder Karte ueber die ueber alle Rollouts gemittelten Endpunkte am
  Rundenende
- Die Karte mit dem hoechsten erwarteten Wert wird als "Lehrer-Aktion"
  gespeichert
- Lookahead-Modus: `full-round-vec` (vektorisiert ueber alle Rollouts; siehe
  [`training/data/vectorized_lookahead.py`](../../training/data/vectorized_lookahead.py))

**Varianten-Verteilung** der Trainingsdaten orientiert sich an der natuerlichen
Ansage-Verteilung des Heuristik-Bots (siehe Eval-Bericht weiter unten). Alle
12 Spielvarianten sind enthalten: 4 x Trumpf, 4 x Gumpf, Oben, Unten,
Slalom-Oben, Slalom-Unten.

## Trainingsverfahren

| Parameter | Wert |
|---|---|
| Verfahren | Behavioral Cloning (Lehrer-Imitation) |
| Loss | Sparse Categorical Crossentropy (Policy) + MSE (Value) |
| Loss-Gewichtung | 1.0 fuer Policy, 0.1 fuer Value |
| Optimizer | Adam, Lernrate 1e-3 |
| Batch-Groesse | 8192 |
| Epochen | 10 (effektiv; mit Early-Stopping bei Val-Loss-Plateau) |
| Shard-Streaming | Ja (tf.data interleave ueber Shard-Pfade; Peak-RAM < 10 GB) |
| Hardware | RTX 3060 12 GB, ~30 Minuten Trainingszeit |

**Trainings-Metriken** (aus `models/mcts_phase2/history.json`):

| Epoche | Val-Loss | Val-Policy-Accuracy |
|---|---|---|
| 1 | 0.973 | 55.4 % |
| 5 | 0.917 | 59.3 % |
| 10 | 0.914 | 59.9 % |

Val-Policy-Accuracy ist die Quote, mit der das Modell dieselbe Karte spielt
wie der MCTS-Lehrer. Wichtiger als die absolute Zahl ist, dass das Modell
auf der Plateau-Loss nicht uebertrainiert.

## Bekannte Schwaechen

1. **Gumpf ist die schwaechste Variante** (54-57 %). Vermutung: Im Gumpf
   bringt das Stich-Lookahead weniger, weil die invertierte
   Nicht-Trumpf-Stoerung die Modellierung schwieriger macht. Verbesserung
   waere ein gezielt auf Gumpf-Stellungen erweiterter Trainingsdatensatz.
2. **Keine Mensch-Validierung**: Win-Rate ist ausschliesslich gegen das
   eigene Vorgaengermodell und gegen die Heuristik gemessen. Wie das
   Modell gegen einen erfahrenen Vorarlberger Jasser spielt, ist nicht
   empirisch belegt.
3. **Greedy-Inference**: Im Eval und in der Web-App spielt das Modell
   greedy (argmax ueber legale Aktionen). Es gibt keine
   Spielstaerken-Diversifikation; gegen denselben Gegner mit denselben
   Karten waehlt es immer dieselbe Aktion. Das macht es vorhersehbar.
4. **Trumpf-Ansage und Weisen werden NICHT vom Modell gespielt** -- diese
   werden vom Heuristik-Bot uebernommen. Die Spielstaerke des Gesamtsystems
   haengt also auch von der Heuristik ab, nicht nur vom NN.
5. **Sehr kleine Stichproben bei Oben** (n=858 Runden). Die 58.7 % sind
   plausibel im Trend, aber statistisch wackliger als die anderen
   Varianten.

## Reproduktion

### Trainingsdaten regenerieren

WSL2 mit GPU:
```bash
# Phase 1: kleine Daten mit v0.5.0 als Lehrer-Init
python -u -m training.data.generate_mcts_data_mp \
    --warm-start models/v5/best.keras \
    --games-per-variant 72 \
    --rollouts-per-card 30 \
    --target 1000 \
    --workers 4 \
    --parallel-threads-per-worker 8 \
    --inference-batch-size 1024 \
    --lookahead-mode full-round-vec \
    --output data/mcts/phase1
```

Phase-2-Aufruf identisch, nur mit `--warm-start models/mcts_phase1/best.keras`
und `--output data/mcts/phase2`. Gesamtlaufzeit auf RTX 3060: ca. 22 Stunden.

### Training

WSL2 oder PowerShell:
```
python -m training.train --data data/mcts/phase1 data/mcts/phase2 --output models/v0.7.0 --epochs 10
```

### Eval gegen Vorgaengermodell

WSL2 oder PowerShell:
```
python -m evaluation.run_eval --a nn --model-a models/v0.7.0/best.keras --b nn --model-b models/v5/best.keras --games 2000 --inference-mode batched-gpu --paired-eval
```
