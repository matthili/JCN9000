## Verwendungszweck

KI-Gegner fuer **Bodensee-Jass** (2 Spieler, jeder mit Hand + sichtbaren +
verdeckten Tisch-Karten) in der Web-App "Heb ab!" (separates Projekt).

**Wichtig:** Dieses Modell ist **nicht austauschbar** mit den Kreuz-Jass-
(v0.7.0) oder Solo-Jass-Modellen (v0.8.0). Bodensee hat:
- 2 statt 4 Spieler
- Eine eigene Encoder-Version (`bodensee_1.0.0`, 291 statt 421 Dimensionen)
- Die Tisch-Mechanik (Hand + sichtbare + verdeckte Karten)

Die Web-App muss anhand der gewaehlten Spielart das passende Modell laden.

**Nicht gedacht fuer:**
- Trumpf-Ansage (uebernimmt der Bodensee-Heuristik-Bot)
- Coaching-/Vorschlag-Funktion ohne klare Kennzeichnung als KI-Empfehlung

## Architektur

|---|---|

## Trainingsdaten

| Quelle | Anzahl Samples |
|---|---|
| Bootstrap (Heuristik vs. Heuristik, ohne MCTS) | ca. 1.667.376 |
| MCTS-Phase 2 (Lehrer-Init: Bootstrap-Modell) | 1.162.080 |
| **Gesamt** | **ca. 2.829.456 Samples** |

**Datenerzeugung:**

Bodensee braucht — anders als Solo — eine eigene Bootstrap-Stufe, weil die
Kreuz/Solo-Modelle den falschen Encoder-Dim haben (421 vs Bodensee 291) und
nicht als Lehrer-Init taugen.

1. **Bootstrap** ([`generate_bodensee_heuristic_data.py`](../../training/data/generate_bodensee_heuristic_data.py)):
   Zwei `BodenseeHeuristicPlayer` spielen ~10.000 Partien gegeneinander. Die
   Heuristik-Karten-Wahl ist das Trainings-Label. Daraus entsteht ein erstes
   Bodensee-NN auf Heuristik-Niveau.
2. **MCTS-Phase 2** ([`generate_bodensee_mcts_data_mp.py`](../../training/data/generate_bodensee_mcts_data_mp.py)):
   Pro Spielposition Monte-Carlo-Lookahead mit Determinisierung der
   verdeckten Karten (eigene + gegnerische + Gegnerhand) und ~30 Rollouts
   pro legaler Karte. Die Karte mit dem hoechsten erwarteten Rundenende-Score
   wird als Lehrer-Aktion gespeichert.

**Variables Spielziel:** 50/50 zwischen 500 und 1000 Punkten pro Partie, damit
das Modell nicht auf ein Spielende-Fenster ueberangepasst ist.

## Trainingsverfahren

| Parameter | Bootstrap | MCTS-Phase 2 |
|---|---|---|
| Verfahren | Behavioral Cloning (Heuristik-Imitation) | Behavioral Cloning (MCTS-Lehrer) |
| Warm-Start | nein (von Grund auf) | ja (aus Bootstrap-Modell) |
| Lernrate | 1e-3 | 5e-4 |
| Batch-Groesse | 1024 | 1024 |
| Epochen | 30 (Early-Stopping bei Epoche 5) | 30 (Verbesserung bis Epoche 25) |
| Hardware | RTX 3060 12 GB | RTX 3060 12 GB |

**Trainings-Metriken Bootstrap:** Val-Policy-Accuracy 95.8 % (= reproduziert
die Heuristik nahezu perfekt -- das ist gewollt fuer einen Bootstrap, keine
echte Spielstaerke). Val-Value-MAE 0.195.

## Bewertung

Eval-Setup: paired-eval (pro Kartenverteilung 2 Partien mit gespiegelten
Sitzen, eliminiert Karten-Glueck), je 4000 Partien, Spielziel-Mix 500/1000.

| Vergleich | v0.9.0 Win-Rate | Gegner Win-Rate |
|---|---|---|
| vs. Bootstrap-Modell (Phase 1) | **74.0 %** | 26.0 % |
| vs. Bodensee-Heuristik | **72.6 %** | 27.4 % |

| Metrik | v0.9.0 | Gegner |
|---|---|---|
| Avg. Score / Partie (vs. Heuristik) | 524.5 | 435.9 |
| Matsch-Rate / Runde (vs. Heuristik) | 0.84 % | 0.10 % |

2-Spieler-Baseline ist 50 %. v0.9.0 liegt 22-24 Prozentpunkte darueber. Bei
4000 paired-Partien betraegt die SD der Win-Rate ca. 0.8 % -- der Abstand zur
Baseline entspricht ca. 28 Standardabweichungen.

**Pro Variante** (Win-Rate v0.9.0 gegen Heuristik):

| Variante | Win-Rate | Stichprobe |
|---|---|---|
| Trumpf Eichel | 71.1 % | n=280 |
| Trumpf Schelle | 69.9 % | n=312 |
| Trumpf Herz | 70.4 % | n=284 |
| Trumpf Laub | 70.4 % | n=284 |
| Gumpf Eichel | 72.7 % | n=414 |
| Gumpf Schelle | 74.0 % | n=308 |
| Gumpf Herz | 74.6 % | n=346 |
| Gumpf Laub | 71.1 % | n=380 |
| Oben | 73.2 % | n=564 |
| Unten | 74.5 % | n=828 |

Alle Varianten zwischen 69.9 % und 74.6 % -- sehr gleichmaessig, kein
Schwachpunkt. Schwaechste: Trumpf Schelle (69.9 %). Staerkste: Gumpf Herz
(74.6 %). Slalom kam in dieser Eval-Stichprobe nicht vor (Heuristik waehlt
es selten an).

## Bekannte Schwaechen

- **Slalom-Varianten nicht empirisch belegt** -- kamen in der Eval-Stichprobe
  nicht vor, weil die Bodensee-Heuristik (die die Ansage macht) Slalom selten
  waehlt.
- **Mensch-vs-Modell-Validierung steht aus** -- die Zahlen sind gegen das
  Bootstrap-Modell und gegen die Heuristik gemessen.
- **`i_am_announcer`-Feature im Encoder** ist im Datengen pragmatisch immer
  auf `False` gesetzt worden. Das Bit ist damit faktisch tot -- wer angesagt
  hat, lernt das Modell nicht. Effekt auf die Spielstaerke ist gemessen
  marginal (Eval-Resultate sind ausgezeichnet), aber eine sauberere
  Implementierung waere fuer eine spaetere Version moeglich.

## Reproduktion

WSL2 mit GPU:
```bash
# Bootstrap-Daten
python -u -m training.data.generate_bodensee_heuristic_data \
    --games 10000 --games-per-shard 100 \
    --output data/bodensee_heuristic_bootstrap

# Bootstrap-Modell trainieren
python -u -m training.train \
    --data data/bodensee_heuristic_bootstrap \
    --output models/bodensee_phase1 --epochs 30

# MCTS-Phase-2-Daten
python -u -m training.data.generate_bodensee_mcts_data_mp \
    --warm-start models/bodensee_phase1/best.keras \
    --games-per-variant 100 --games-per-chunk 25 --rollouts-per-card 30 \
    --workers 8 --parallel-threads-per-worker 32 \
    --output data/bodensee_mcts/phase2

# Phase-2-Modell trainieren
python -u -m training.train \
    --data data/bodensee_mcts/phase2 \
    --warm-start models/bodensee_phase1/best.keras \
    --output models/bodensee_phase2 --epochs 30 --learning-rate 5e-4

# Eval
python -m evaluation.run_bodensee_eval \
    --a nn --model-a models/bodensee_phase2/best.keras \
    --b heuristic --games 4000 --paired-eval --inference-mode batched-gpu
```
