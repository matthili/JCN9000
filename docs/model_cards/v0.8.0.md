## Verwendungszweck

KI-Gegner fuer **Solo-Jass** (4 Spieler, jeder gegen jeden) in der Web-App
"Heb ab!" (separates Projekt). Trifft pro Spielposition eine Karten-
Entscheidung basierend auf der eigenen Hand und der bekannten Spielhistorie.

**Wichtig:** Dieses Modell ist **nicht austauschbar** mit dem v0.7.0-Modell
fuer Kreuz-Jass. Die Reward-Struktur ist fundamental anders (eigene Punkte
statt Team-Punkte), die Schmieren-/Sparen-Strategie ist invertiert. Die
Web-App muss anhand der gewaehlten Spielart das passende Modell laden.

**Nicht gedacht fuer:**
- Trumpf-Ansage (wird vom Solo-Heuristik-Bot uebernommen)
- Weisen-Wahl (Heuristik)
- Coaching-/Vorschlag-Funktion ohne klare Kennzeichnung als KI-Empfehlung

## Architektur

| Eigenschaft | Wert |
|---|---|
| Modell-Typ | Mehrschichtiges Perzeptron (MLP) |
| Hidden-Layer | 768 - 768 - 384 (ReLU) |

| Ausgabe | `policy` (36 Wahrscheinlichkeiten) + `value` (1 Skalar in `[-1, 1]`) |
| Custom-Layer | `MaskBias` (gross negativer Bias auf illegale Aktionen vor Softmax) |
| Framework | TensorFlow / Keras |
| Browser-Format | TensorFlow.js |

**Architektur ist identisch zum v0.7.0-Modell.** Nur die Gewichte sind
spezifisch fuer Solo-Jass trainiert.

**Datenerzeugung** (siehe
[`training/data/generate_solo_mcts_data_mp.py`](../../training/data/generate_solo_mcts_data_mp.py)):

Pro Spielposition wurde ein Monte-Carlo-Lookahead mit Solo-Reward durchgefuehrt:
- Pro legaler Karte ca. 30 Rollouts mit determinisierten Mitspieler-Haenden
- Bewertung: eigene Endpunkte minus staerkster Gegner-Endpunkt (statt
  Team-Differenz wie bei Kreuz-Jass)
- Die Karte mit dem hoechsten erwarteten Wert wird als Lehrer-Aktion
  gespeichert
- Lookahead-Modus: `full-round-vec` (vektorisiert ueber alle Rollouts)

**Variables Spielziel:** Die Trainings-Partien wurden mit unterschiedlichen
Spielzielen gespielt (Default 50/50 zwischen 500 und 1000 Punkten), damit das
Modell nicht auf ein bestimmtes Spielende-Fenster ueberangepasst ist. Score-
Features sind normalisiert (`score / target_score`), daher sieht das Modell
"Halbzeit" als 0.5 unabhaengig vom konkreten Ziel.

**Solo-spezifische Pipeline:** Im Solo gibt es keinen Team-Partner. Die
Heuristik wurde entsprechend angepasst (`SoloHeuristicPlayer`) -- kein
Schmieren mehr, aggressivere Eigen-Stiche.

## Reproduktion

### Trainingsdaten regenerieren (WSL2 mit GPU)

```bash
# Solo-Phase 1 (Lehrer-Init: v5-Kreuz-Jass-Modell als neutraler Start)
python -u -m training.data.generate_solo_mcts_data_mp \
    --warm-start models/v5/best.keras \
    --games-per-variant 100 \
    --rollouts-per-card 30 \
    --target-distribution "500:0.5,1000:0.5" \
    --workers 8 --parallel-threads-per-worker 32 \
    --output data/solo_mcts/phase1

# Solo-Phase 2 (Lehrer-Init: das in Phase 1 trainierte Modell)
python -u -m training.data.generate_solo_mcts_data_mp \
    --warm-start models/solo_phase1/best.keras \
    --games-per-variant 500 \
    --rollouts-per-card 30 \
    --target-distribution "500:0.5,1000:0.5" \
    --workers 8 --parallel-threads-per-worker 32 \
    --output data/solo_mcts/phase2
```

### Training

```bash
# Phase 1
python -u -m training.train \
    --data data/solo_mcts/phase1 \
    --output models/solo_phase1 \
    --epochs 20

# Phase 2 (Warm-Start aus Phase 1)
python -u -m training.train \
    --data data/solo_mcts/phase2 \
    --warm-start models/solo_phase1/best.keras \
    --output models/solo_phase2 \
    --epochs 15 \
    --learning-rate 5e-4
```

### Eval

```bash
python -m evaluation.run_solo_eval \
    --a nn --model-a models/solo_phase2/best.keras \
    --b nn --model-b models/solo_phase1/best.keras \
    --games 800 \
    --paired-eval
```
